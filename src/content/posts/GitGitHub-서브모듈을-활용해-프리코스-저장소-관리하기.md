## 프리코스가 종료되고..
4주 동안의 프리코스를 모두 마무리하고 본격적으로 복습하기 전에 내가 작성했던 코드들을 한눈에 모아 보는 저장소가 필요하다고 생각했다. 저장소 안의 코드를 일일이 붙여넣기 하려다가 더 좋은 방법이 있을 것 같아 gpt에게 물어보니 git의 서브모듈(Submodule)이라는 기능이 있다는 것을 알려줘 이를 사용해 보기로 했다.

<br/>

## 서브모듈이란?

서브모듈이란 **Git 저장소 안에 다른 Git 저장소를 프로젝트 하위 디렉토리로 추가하는 기능**이다. 각 서브모듈은 독립적인 Git 저장소로서 관리되며, 부모 프로젝트에서는 서브모듈의 특정 커밋을 참조한다. 따라서 서브모듈을 사용하면 **다른 프로젝트의 코드를 포함하면서도 해당 프로젝트와 독립적으로 개발 및 업데이트 할 수 있다**.

이런 기능이 왜 필요한 걸까? [git SCM](https://git-scm.com/book/ko/v2/Git-%EB%8F%84%EA%B5%AC-%EC%84%9C%EB%B8%8C%EB%AA%A8%EB%93%88)에서 이유를 찾아 봤다. 프로젝트를 수행하다 보면 다른 프로젝트를 사용해야 하는 경우가 종종 있다. 이는 외부에서 개발한 라이브러리일 수도 있고, 내부의 다른 프로젝트에서 공통으로 사용하는 라이브러리일 수도 있다. 이를 사용할 때는 Shared 라이브러리 형태로 쓰거나 직접 그 소스코드를 복사할 수도 있는데, Shared 라이브러리를 사용하면 프로젝트에 맞게 수정해서 사용하기 어렵다는 단점이 있고, 소스코드를 직접 복사하는 경우 라이브러리가 업데이트됐을 때 merge 하기가 어렵다는 단점이 있다고 한다.

서브모듈은 이러한 문제를 해결해 각 저장소의 커밋을 독립적으로 관리하면서도 필요한 경우 쉽게 merge 할 수 있도록 한다.

## 프리코스 코드 추가하기
이제 서브모듈로 저장소에 내 프리코스 코드들을 추가해 보자. 

### 서브모듈 추가
아래의 명령어를 사용해서 1주차 저장소를 모듈로 가져온다. (나는 제출한 코드를 모아 둘 `submit` 디렉터리를 만든 후 진행했다.)

```
git submodule add <1주차 저장소 URL>
```

위 명령어를 입력하면 서브디렉토리와 하위 프로젝트 URL의 매핑 정보를 담은 `.gitmodules` 파일과 `javascript-baseball-6` 프로젝트 폴더가 생성된다. 그리고 해당 저장소의 코드들을 VScode에서 확인할 수 있다!

```
[submodule "submit/javascript-baseball-6"]
	path = submit/javascript-baseball-6
	url = https://github.com/llqqssttyy/javascript-baseball-6.git
```

![vscode에서 보이는 파일들](/posts/GitGitHub-서브모듈을-활용해-프리코스-저장소-관리하기/image-01.webp)

위의 변경사항을 커밋하면 재밌는 사실을 발견할 수 있다. 

```
git commit -am '커밋 메세지'
```

![](/posts/GitGitHub-서브모듈을-활용해-프리코스-저장소-관리하기/image-02.webp)

커밋이 성공적으로 진행되면 나오는 메세지에서 .gitmodules와 서브모듈 디렉토리의 디렉토리 모드가 다르게 표시된다는 것을 발견할 수 있다.

git은 보통의 파일은 `100644`, 실행파일은 `100755`라는 디렉토리 모드를 사용해 관리한다. 위 이미지를 보면 .gitmodules는 `100644`를 사용하는 반면 서브모듈은 `160000`으로 관리되는 것을 알 수 있다. 이는 서브모듈이 일반적인 파일이나 디렉토리가 아니라 특별한 커밋으로 취급된다는 것을 의미한다. 
그렇다. 이 커밋의 정체는 바로 서브모듈 저장소의 커밋 해쉬 링크다.

변경 사항을 원격 저장소에 push하고 해당 저장소를 보면 위의 내용을 직관적으로 이해할 수 있다. submit 디렉토리로 들어가 보면, 서브모듈 디렉토리 이름으로 된 링크를 확인할 수 있다. 해당 링크를 클릭하면 서브모듈이 가리키고 있는 버전(커밋)으로 이동한다.
![](/posts/GitGitHub-서브모듈을-활용해-프리코스-저장소-관리하기/image-03.webp)

### 브랜치 변경
아직 끝나지 않았다! 서브모듈을 추가하면 기본적으로 main(혹은 master) 브랜치로 클론되기 때문에 서브모듈 디렉터리가 가리키는 커밋을 미션을 진행한 브랜치의 최신 커밋으로 바꿔줘야 한다. 
그리고 다시 부모 저장소로 돌아와 해당 변경사항을 커밋 후 원격 저장소에 push 한다.

```
cd submit/javascript-baseball-6
git checkout <본인 아이디>

cd ../
git commit -am '커밋 메시지' 
git push origin main 
```

위 과정을 반복해서 모든 저장소를 서브모듈로 연결시킨다.

```
git submodule add <저장소 URL>

cd <서브모듈 디렉토리>
git checkout <본인 아이디>

cd <부모 저장소 디렉토리>
git commit -am '커밋 메시지' 
git push origin main 
```

### (추가) 브랜치를 지정하여 추가하기
최종 코딩 테스트 저장소도 서브 모듈로 추가하는 중에 checkout으로 브랜치를 바꿔줬음에도 원격 저장소에서 반영이 되지 않는 문제가 나타났다.

그래서 기존에 추가했던 모듈을 지우고 브랜치를 지정하여 다시 추가하는 방식을 알아 보았다.
```
# 브랜치를 지정하여 서브모듈로 추가하기
git submodule add -b <브랜치 이름> --force <저장소 URL> <서브모듈 디렉토리 이름>
```

기존에 있던 서브모듈을 삭제할 때 `git submodule deinit -f`만 사용하고 .git/modules에서 삭제하는 처리를 하지 않아 --force 옵션을 붙일 수 밖에 없었다. 서브모듈을 올바르게 삭제하는 방법은 아래와 같다. ([참고](http://snowdeer.github.io/git/2018/08/01/how-to-remove-git-submodule/))

```
git submodule deinit -f <서브모듈 디렉토리 이름>
rm -rf .git/modules/<서브모듈 디렉토리 이름>
git rm -f <서브모듈 디렉토리 이름>
```

## 부모 프로젝트를 클론할 때

위에서 언급했듯 서브모듈들은 모두 '특정 커밋을 가리키고 있는 커밋'이다. 따라서 다른 컴퓨터에서 저장소를 클론 하면 **서브모듈 디렉토리는 텅 빈 채 클론** 된다. 따라서 클론 시 서브모듈의 코드까지 받아오고 싶다면 --recurse-submodules 옵션을 넣어줘야 한다.

```
git clone --recurse-submodules <레포지터리 url>
```
