## 〰️ 캐시 관리 전략 수립 시 고려할 점
캐시 관리 전략을 수립한다는 것은 캐시의 유효 기간을 정하고, 적절한 시점에 무효화하는 것을 의미합니다.

### 조건부 요청

사실 브라우저는 **캐싱된 리소스가 만료**되었거나, **검증 헤더(ETag, Last-Modified)가 제공**된 경우에 자동으로 **조건부 요청(Conditional Request)**을 보내 캐시된 자원의 유효성을 확인합니다.
조건부 요청은 서버에 자원이 변경되었는지 확인하고, 변경 여부에 따라 다음과 같은 응답을 반환합니다.

- `304 Not Modified`: 자원이 변경되지 않아 기존 캐시를 그대로 사용.
- `200 OK`: 자원이 변경되어 새로운 데이터를 반환.

그렇다면 조건부 요청이 자동으로 캐시를 관리하는데, 왜 추가적인 무효화 전략이 필요할까요?
조건부 요청만으로는 실시간 데이터 제공, 명시적 제어, 자원 변경 주기 관리와 같은 요구사항을 모두 충족하지 못하기 때문입니다. 이를 해결하기 위해 아래와 같은 무효화 전략이 사용됩니다.

### 캐시 무효화 전략

캐시 무효화 전략에는 아래의 방법들이 있습니다.

- **시간 기반 캐시 무효화**(Time-based Invalidation)
- **버전 기반 캐시 무효화**(Version-based Invalidation)
- **수동 캐시 무효화**(Manual Invalidation)

#### 1. 시간 기반 캐시 무효화

이름 그대로 자원의 유효 기간을 설정해 기간이 만료되면 자동으로 새로운 정보를 받아오도록 하는 전략입니다.
`Cache-Control` 및 `Expires` 헤더의 `max-age` 디렉티브 등을 사용해 지정할 수 있습니다.

이 방법은 **업데이트 주기가 일정한 데이터**에 적합하며, 비교적 **구현이 간단하다**는 장점이 있습니다. 
S3를 사용하는 경우 자원에 metadata로 `Cache-Control` 헤더를 추가해 간단히 구현이 가능합니다.
하지만 실시간으로 업데이트 되는 데이터엔 적합하지 않으며, 자주 변경되는 데이터를 다룰 때 주의가 필요합니다.

#### 2. 버전 기반 캐시 무효화

자원의 URL이나 **파일 이름에 버전을 포함**시켜, 자원 업데이트 시 캐시된 데이터를 무효화하고 새로운 버전의 데이터를 캐싱하는 전략입니다.
파일 빌드 시 해시값을 붙여 자원의 이름을 변경하거나, 서버에서 ETag를 사용해 구현이 가능합니다.

이 방법은 자원이 변경될 때마다 캐시를 자동으로 무효화하므로, **최신 데이터를 보장**할 수 있습니다. 매번 자원의 이름이 변경되기 때문에, 이를 참조하는 HTML 파일의 업데이트가 필요하지만, 번들러를 사용한다면 자동으로 갱신됩니다.

#### 3. 수동 캐시 무효화

CloudFront의 캐시를 수동으로 invalidation하거나, 브라우저 캐시를 수동으로 삭제하는 것을 의미합니다. 사용자에게 찾아가 브라우저의 캐시를 invalidation 하는 일은 없겠지만(ㅎㅎ) 원본 서버(예: S3)의 리소스가 변경되었을 때 CloudFront가 저장하고 있는 캐시를 지우기 위해 수동 invalidate하기도 합니다. 크루루의 CD 플로우에도 이와 같은 과정이 있습니다.

지금까지의 내용을 표로 정리하자면, 다음과 같습니다.

| **전략**                  | **설명**                                                                                  | **장점**                                                                                  | **단점**                                                                                  | **적용 사례**                                                                                     |
|---------------------------|-------------------------------------------------------------------------------------------|------------------------------------------------------------------------------------------|------------------------------------------------------------------------------------------|---------------------------------------------------------------------------------------------------|
| **시간 기반 캐시 무효화** (Time-based) | 자원의 유효 기간을 설정하여, 만료되면 새로운 정보를 받아오도록 하는 전략. <br>`Cache-Control` 및 `Expires` 헤더의 `max-age` 디렉티브 등을 사용. | - **구현이 간단**하여 적용하기 쉬움. <br>- **업데이트 주기가 일정한 데이터**에 적합.       | - 실시간 데이터에는 부적합.<br>- 자주 변경되는 데이터의 경우 잘못된 정보 제공 위험.       | 하루 한 번 업데이트되는 뉴스 페이지, 주기적으로 갱신되는 API 데이터.                              |
| **버전 기반 캐시 무효화** (Version-based) | 자원의 URL이나 파일 이름에 **버전(또는 해시)**을 포함하여, 자원이 변경될 때마다 새로운 버전의 자원을 캐싱. <br>서버에서 ETag를 사용하거나 빌드 시 해시값을 추가해 구현 가능. | - 자원이 변경될 때마다 **최신 데이터를 보장**. <br>- 조건부 요청 없이 무효화 가능.       | - 자원 이름 변경 시 이를 참조하는 HTML 파일도 업데이트 필요. <br>- 번들러가 없다면 번거로움. | 번들러(Webpack, Vite)를 사용해 해시를 붙인 JS/CSS 파일.<br>버전 업데이트가 중요한 SPA.            |

## 〰️ 크루루의 캐시 관리 전략

크루루는 **리소스 별 변경 빈도와 일관성**에 따라 캐시를 다르게 관리하고 있습니다.

### 리소스 별 캐싱 설정

| 구분          | 브라우저 캐시 | CDN 캐시     | 비고                                                          |
| ------------- | ------------- | ------------ | ------------------------------------------------------------- |
| HTML          | 0             | 1년          | Cache invalidation 활용 여부에 따라 CDN 캐시 기한을 조정 |
| CSS/JS        | 1년           | 1년          | 파일명을 이용한 캐시 버스팅 활용                              |
| Image         | 1년           | 1년          | 파일명을 이용한 캐시 버스팅 활용                              |
| Etc (Default) | 1일           | (기본값 1일) |                                                               |

- **HTML 리소스**: 항상 최신 리소스를 제공하기 위해 브라우저 캐시는 max-age=0으로 설정하고, CDN은 최대 1년 동안 캐싱되도록 설정했습니다. index.html에 변경이 있을 경우 cd 과정에서 CloudFront의 캐시를 무효화 해주는 과정을 거치므로 사용자는 최신 버전의 index.html을 볼 수 있습니다.
- **CSS/JS 및 이미지 리소스**: 파일명에 해시 값을 추가하여 캐시 버스팅을 활용하고, 브라우저와 CDN 캐시 모두 최대 1년 동안 캐시를 유지합니다.
- **기타 리소스**: 이미지와 같은 기타 리소스는 기본 캐싱 설정인 1일동안 유지됩니다.

### 캐시 정책 적용

위와 같은 캐시 정책을 바탕으로 **CloudFront의 Function**을 이용해 요청 별로 Cache-Control 헤더를 동적으로 적용하도록 처리했습니다. 이렇게 하면 다양한 리소스 유형에 유연하게 대응할 수 있습니다.

이러한 정책을 CloudFront의 Function을 사용해 구현하게 된 결정엔 몇 가지 고려사항도 있습니다.

- CloudFront Function의 실행 비용: CloudFront Function은 호출 건당 비용이 발생하므로, 캐시 설정을 적절하게 유지하여 불필요한 캐시 무효화를 방지하고, 서버 부하를 최소화해야 합니다.
- 응답 헤더 정책 제한: 우아한테크코스 프로젝트 환경에서는 각 팀당 생성할 수 있는 응답 헤더 정책이 1개로 한정되어 있습니다. 따라서 하나의 Function으로 다양한 리소스에 대한 캐시 정책을 관리하도록 했습니다.

이로써 사용자에게 최신 콘텐츠를 빠르고 안정적으로 제공할 수 있는 캐싱 정책을 반영할 수 있었습니다.

이번 글에서는 웹 캐시의 개념과 필요성, 그리고 크루루의 캐싱 정책을 설정하고 CloudFront Function을 통해 이를 구현한 방법을 소개했습니다.

크루루에 캐싱 정책을 설정하고 invalidation하는 과정에서 캐시의 다양한 적용 방식과 고려 사항을 깊이 있게 배울 수 있었습니다. 특히 이러한 정책이 서비스 요구사항에 따라 유연하게 조정될 수 있음을 깨달았습니다.
