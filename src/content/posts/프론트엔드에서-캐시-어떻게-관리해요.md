제목에 쓰인 질문을 반복해볼게요. 프론트엔드에서 캐시, 어떻게 그리고 왜 쓰는 건가요?

희미하지만 전공 수업에서 캐시를 배울 때가 기억이 납니다. 위키피디아에 적힌대로 배웠던 것 같아요.

> **Cache (computing)**
> In computing, a cache (/kæʃ/ ⓘ KASH)[1] is a hardware or software component that stores data so that future requests for that data can be served faster; the data stored in a cache might be the result of an earlier computation or a copy of data stored elsewhere.
>
> 컴퓨팅에서 캐시(/kæʃ/, KASH)는 하드웨어 또는 소프트웨어 구성 요소로, 데이터를 저장하여 이후 해당 데이터에 대한 요청을 더 빠르게 처리할 수 있도록 합니다. 캐시에 저장된 데이터는 이전에 수행된 연산 결과이거나 다른 곳에 저장된 데이터의 복사본일 수 있습니다.

복사본, 빠르게, 처리.. 키워드는 알겠지만 여전히 추상적이죠.

그러던 중 프로젝트를 CloudFront로 배포하며 CDN 캐시에 대해서 알게 되었고, 이를 시작으로 웹 서비스, 특히 프론트엔드 단에서 캐시가 어떻게 동작할 수 있는지 알아보기 시작했습니다.

이번 글에서는 웹 서비스 안에서 **캐시의 기본 개념**과 **캐시를 관리하는 헤더**들, 그리고 **주의해야 할 점**까지 다루겠습니다.

## 〰️ 캐시는 어디에 저장되고 어디에서 활용하나요?

> 데이터를 한 번 받아오거나 계산한 결과를 임시로 저장해두고, 다음에 같은 데이터를 요청할 때 저장된 값을 재사용하는 매커니즘. 브라우저, 서버, 네트워크 계층 등 다양한 위치에서 활용된다.

예시를 들어보겠습니다.
웹 페이지를 처음 로드할 때 이미지를 다운로드했다면 브라우저는 이를 캐시에 저장해둡니다.
이후 같은 이미지를 다시 요청하면 서버에서 새로 다운로드하지 않고, 캐시에서 가져와 훨씬 빠르게 화면에 표시합니다.
서버의 경우엔 데이터베이스의 결과를 저장하여, 요청이 들어올 때마다 반복적인 무거운 연산을 수행하지 않고도 빠르게 응답하는 것도 캐싱의 예시가 될 수 있습니다.

### 캐시의 종류

캐시는 저장되는 위치에 따라 크게 두 가지로 나눌 수 있습니다.

- **사설 캐시(private cache)**
  브라우저나 개인 디바이스에서 데이터를 저장해두는 캐시입니다.

- **공용 캐시(public cache)**:
  여러 사용자와 공유하는 캐시로, 주로 CDN(Content Delivery Network) 서버나 프록시 서버에서 데이터를 저장합니다.

이 글은 사설 캐시인 브라우저 캐시와, 공용 캐시인 CDN 캐시에 대해서 다루고 있습니다.

## 〰️ 브라우저 캐시 사용하기

브라우저는 한번 캐시를 저장하면 유효 기간이 지나거나 강제로 비우기 전까지 캐시를 계속 가지고 있습니다. 따라서 브라우저 캐시를 관리한다는 것은 만료 기간을 관리한다는 것과 같습니다.

그럼 캐시의 유효 기간은 어떻게 지정해줄 수 있을까요? 이를 위해 사용하는 것이 바로 `Cache-Control` 헤더입니다.

### 1. `Cache-Control` header

이름 그대로 캐시를 제어하기 위한 헤더입니다. 이때 헤더의 값으로 들어가 캐싱 동작을 제어하는 명령어를 **'디렉티브'**라고 합니다. 디렉티브로는 캐시 유효 기간 외에도 무효화 조건, 저장 여부 등을 결정할 수 있습니다.

#### 디렉티브의 종류

디렉티브는 크게 아래와 같이 분류할 수 있습니다.

- **캐시 요청 디렉티브**:
  - HTTP 요청 내에서 **클라이언트에 의해 사용**될 수 있는 표준 Cache-Control 디렉티브. 요청을 보낼 때 캐시된 데이터를 사용할지 무시할지를 결정하는 기준이 된다.
  - (e.g. max-age, no-cache 등)
- **캐시 응답 디렉티브:**
  - HTTP 응답 내에서 **서버에 의해 사용**될 수 있는 표준 Cache-Control 디렉티브. 해당 응답이 클라이언트와 공유 캐시에 얼마나 오래 저장될지 결정한다.
  - (e.g. public, private, s-maxage 등)
- **확장 디렉티브**:
  - 핵심 HTTP 캐싱 표준 문서에 속하지 않는 디렉티브.
  - (e.g. immutable)

#### 주요 디렉티브

- `max-age`
  - 캐시가 유효한 기간을 초 단위로 설정.
  - 요청과 응답에서 모두 사용될 수 있다.
  - 응답에 설정되면 클라이언트는 설정 시간동안 캐시된 데이터를 사용한다.
- `no-cache`
  - 캐시가 존재하더라도 항상 서버에 데이터 유효성 검증을 요청한다.
  - 유효성 검증은 `ETag`나 `Last-Modified` 등의 헤더를 사용한다.
- `no-store`
  - 리소스를 캐시에 저장하지 않고 매번 서버에 새로 데이터를 받아오도록 한다.
  - 민감한 데이터나 자주 변경되는 데이터에 적합하다.

#### 디렉티브의 독립성

요청 디렉티브와 응답 디렉티브는 서로 독립적으로 동작한다는 특성을 가지고 있습니다.
이는 **클라이언트가 요청을 보낼 때 설정하는 Cache-Control 디렉티브가 서버 응답에 적용된다는 보장은 없으며, 서버는 요청과 별개로 자신의 캐싱 정책을 따를 수 있다**는 걸 의미합니다.

예를 들어 다음과 같은 상황에서 디렉티브는 아래와 같이 적용됩니다.

> - 클라이언트는 요청 디렉티브로 `Cache-Control: no-cache`를 설정한다.
> - 서버는 응답 디렉티브로 `Cache-Control: max-age=3600`을 설정한다.

1. **클라이언트 최초 요청**
   - 캐시가 없는 상태에서 최초 요청을 보낸다.
2. **서버의 최초 응답**
   - 서버가 보낸 응답은 서버의 캐시 정책(`max-age=3600`)에 따라 캐시가 브라우저에 3600초동안 저장된다.
3. **이후 요청**
   - 클라이언트 디렉티브로 `no-cache`가 설정되어 있기 때문에 캐시된 데이터가 있어도 유효성 검사를 위해 항상 서버로 요청을 보낸다.
   - 서버가 “수정되지 않음”을 나타내는 `304 Not Modified` 응답을 보내면, 클라이언트는 기존 캐시 데이터를 사용한다.

위 예시처럼, 클라이언트가 설정한 디렉티브는 서버의 디렉티브에 영향을 주지 않습니다.
**브라우저는 서버의 디렉티브에 따라 데이터를 캐시에 저장**하지만, **이 데이터를 사용할지 여부는 클라이언트 디렉티브에 의해 결정**됩니다. 이러한 특성을 **디렉티브의 독립성**이라고 합니다.

### 2. 다른 헤더들

`Cache-Control` 헤더 외에도, 캐시 관리에 사용되는 헤더들이 있습니다. 이는 간략하게 다루겠습니다.

#### Expires 헤더

> 캐시 만료 시간을 명시적으로 설정하는 헤더.

- timestamp가 들어가게 된다. (e.g. `Expires: Wed, 21 Oct 2015 07:28:00 GMT`)
- Cache-Control의 max-age, s-maxage에 의해 덮어씌워질 수 있다.

#### ETag 헤더

> 리소스의 고유 식별자를 생성해 변경 여부를 판단. 파일의 콘텐츠가 변경될 때마다 새로운 ETag 값이 생성된다.

**동작 방식**

- 서버는 리소스를 응답할 때 ETag 헤더와 함께 식별자를 포함해 클라이언트에 전달한다.
- 클라이언트는 이후 동일 리소스를 요청할 때 `If-None-Match` 헤더에 ETag 값을 포함해 서버에 보낸다.
- 서버는 ETag 값을 확인해 리소스가 변경되지 않았다면 `304 Not Modified` 응답을 반환하고, 변경된 경우 새로운 데이터를 전달한다.
- 해시 계산이라는 추가 연산이 필요하기 때문에 서버에 부담이 될 수 있다.

#### Last-Modified 헤더

> 리소스의 최종 수정 시간을 기준으로 캐시 유효성을 판단한다.

- Last-Modified는 상대적으로 변경이 적은 리소스나 성능이 중요한 상황에서 사용 가능하다.
- 수정 시간에 의존하기 때문에 1초 이내의 변경이나 **시간대 차이로 인한 오차가 발생**할 수 있다.
- ETag와 Last-Modified가 모두 있는 경우 ETag가 사용된다.

## 〰️ CloudFront 캐시 사용하기

### CDN이란?

> 글로벌 네트워크에 콘텐츠를 분산해 가까운 위치에서 데이터를 제공함으로써 웹 페이지 로딩 속도를 향상시킨다.

CDN과 클라이언트 사이의 자원 요청은 일반 서버와 동일하게 `Cache-Control` 헤더를 사용할 수 있습니다.
다만 CDN은 원본 서버와 클라이언트 사이에 놓인 edge 서버이므로, 멀리 위치한 오리진 서버(원본 서버)와 소통을 최소화하기 위해 캐시를 사용하고 있습니다.

AWS의 대표적인 CDN 서비스인 CloudFront 또한 캐싱을 사용하고 있습니다. 자세한 내용은 아래와 같습니다.

### CloudFront의 원본 리소스 캐싱

CloudFront는 원본 리소스를 캐싱합니다.
만약 오리진 서버에서 Cache-Control이나 Expires 헤더를 설정하지 않았다면, CloudFront는 기본 캐싱 동작을 따릅니다.

- CloudFront는 기본적으로 **각 객체를 24시간(86,400초) 동안 캐싱**합니다.
- 이 기본값은 CloudFront 배포의 캐시 동작 설정에서 최소 TTL(Minimum TTL), 기본 TTL(Default TTL), 최대 TTL(Maximum TTL) 값을 조정하여 변경할 수 있습니다.

기본적으로 24시간동안 캐싱하기 때문에 원본 리소스(S3 객체)를 업데이트한 결과를 즉각 반영하고 싶다면 cache를 invalidation 해주어야 합니다.

### CloudFront의 응답 리소스 캐싱

CloudFront로부터 받은 리소스를 브라우저에 캐싱하게 만들고 싶다면 아래의 방법을 사용할 수 있습니다.

1. **오리진 서버에서 Cache-Control 헤더 설정** ([출처](https://docs.aws.amazon.com/ko_kr/AmazonCloudFront/latest/DeveloperGuide/Expiration.html))
   가장 간단한 방법은 오리진 서버(Amazon S3, EC2 등)에서 직접 Cache-Control 헤더를 설정하는 것입니다. 이렇게 하면 CloudFront는 오리진 서버의 응답 헤더를 그대로 클라이언트에 전달합니다.
   Amazon S3의 경우 S3 콘솔에서 객체를 선택한 후, '메타 데이터'를 편집해 Cache-Control 헤더를 추가할 수 있습니다.

2. **CloudFront 응답 헤더 정책 사용** ([출처](https://docs.aws.amazon.com/ko_kr/AmazonCloudFront/latest/DeveloperGuide/modifying-response-headers.html))
   오리진 서버를 수정하기 어려운 경우, CloudFront의 응답 헤더 정책을 활용하여 Cache-Control 헤더를 추가할 수 있습니다.

3. **CloudFront Functions 또는 Lambda@Edge 사용** ([출처](https://docs.aws.amazon.com/ko_kr/AmazonCloudFront/latest/DeveloperGuide/example-function-add-cache-control-header.html))
   더 복잡한 조건이나 동적 헤더 설정이 필요한 경우, CloudFront Functions나 Lambda@Edge를 사용하여 응답 헤더를 수정할 수 있습니다.
   > - **CloudFront Functions**
   >   JavaScript로 작성된 간단한 함수로, 요청 또는 응답을 처리할 수 있습니다.
   >   응답 이벤트에서 Cache-Control 헤더를 추가하거나 수정할 수 있습니다.
   > - **Lambda@Edge**
   >   더 복잡한 로직이 필요한 경우 AWS Lambda를 사용하여 응답을 처리할 수 있습니다.

AWS에서는 캐싱 정책을 반영할 다양한 방법을 제공하고 있습니다. 팀의 상황과 요구사항에 맞는 방법으로 적용하면 됩니다. (크루루는 CloudFront Functions을 사용해 구현하고 있습니다. 이는 다른 포스팅에서 다루겠습니다.)

## 캐시의 장단점

다양한 계층에서 캐시를 사용할 수 있지만 이들을 모두 관통하는 목적은 **성능 향상**입니다.
API 요청 수를 줄이고 서버의 처리 속도를 개선함으로써 사용자에게 더 빠르고 원활한 서비스를 제공하기 위해서입니다.

하지만 캐시는 만능 도구가 아닙니다.
잘못 관리할 경우 사용자가 최신 데이터를 확인하지 못하거나, 과도한 메모리 사용으로 시스템 자원을 낭비할 수 있으며, 민감한 데이터를 저장하면 보안 취약점으로 이어질 수도 있습니다.
따라서 캐시는 데이터의 특성과 사용 목적에 따라 신중하게 관리되어야 합니다. 이에 대해서는 이어지는 포스팅에서 다루도록 하겠습니다.
