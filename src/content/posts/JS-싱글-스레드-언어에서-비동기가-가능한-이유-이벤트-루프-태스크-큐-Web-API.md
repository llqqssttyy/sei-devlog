## 자바스크립트에서 비동기 처리가 필요한 이유?

자바스크립트는 **싱글 스레드**로 동작하도록 설계되었습니다. 이는 자바스크립트 코드를 파싱하고 실행하는 엔진이 **단 하나의 콜 스택을 사용하며 한 번에 오직 하나의 작업만 처리**할 수 있음을 의미합니다. 이 동기적 처리 방식은 코드 실행 순서를 명확하게 보장하는 장점이 있지만 단점도 존재합니다. 만약 오래 걸리는 작업(예: 복잡한 연산, 네트워크 요청)이 발생하면 그동안 다른 모든 작업이 중단되어 프로그램이 **블로킹 상태**에 빠질 수 있습니다.

예를 들어 웹 페이지에서 대용량 데이터를 처리하거나 서버에서 응답을 기다린다고 가정해봅시다. 그 시간 동안 사용자의 클릭, 스크롤, 입력 등 모든 상호작용이 막힐 것입니다. 웹 페이지가 잠시 멈춘 것처럼 보이게 되고 사용자 경험은 크게 저하됩니다.


<br/>

## 비동기 코드 처리 과정

이런 자바스크립트 엔진의 단점을 보완하기 위해 **자바스크립트 런타임 환경**(브라우저, Node.js)는 비동기 처리를 지원합니다. 그리고 이를 가능하게 하는 것이 바로 **이벤트 루프**와 **태스크 큐**, 그리고 **Web API**입니다.

비동기 코드를 처리하는 기본 과정은 다음과 같습니다.

- 동기적 코드는 자바스크립트 엔진의 **콜 스택**에서 바로 처리됩니다.
- 비동기 작업(타이머 등)은 **Web API**로 보내져 실행됩니다.
- 완료된 비동기 작업의 콜백은 **태스크 큐**에서 대기합니다.
- **이벤트 루프**는 콜 스택이 비어 있으면 **태스크 큐**에서 대기 중인 작업을 **우선순위**에 따라 **콜 스택**에 넣어 실행합니다.(여기서 중요한 것은 우선순위입니다. 이는 [이벤트 루프](#이벤트-루프)에서 더 자세히 다룹니다.)

이 동작을 기억해 두고, 이러한 실행 환경을 구성하는 요소들을 하나씩 살펴봅시다.

## 1️⃣ Web API

**Web API**는 브라우저가 제공하는 API로, 자바스크립트 엔진이 직접 처리할 수 없는 비동기 작업(예: 타이머, 네트워크 요청 등)을 관리합니다. 이러한 비동기 작업이 완료되면, Web API는 이벤트 루프와 태스크 큐를 통해 해당 작업의 콜백을 실행합니다.

### 사례 1 - `setTimeout`

자바스크립트가 setTimeout을 실행할 때, Web API가 어떤 역할을 하는지 살펴보겠습니다.

```js
setTimeout(() => {
  console.log("timer finished");
}, 2000);
```

#### 실행 과정에서 Web API의 역할

1. **콜 스택에 setTimeout 함수가 push된다.** 자바스크립트 엔진에서 setTimeout() 함수가 호출되면, 이 함수의 실행이 콜 스택에서 시작됩니다.

2. **setTimeout 함수가 실행되면 Web API로 타이머와 콜백이 전달된다.** setTimeout() 함수는 Web API에게 타이머와 콜백 함수를 전달합니다. 자바스크립트 엔진은 setTimeout을 실행하고 **콜 스택에서 제거(pop)**됩니다.

3. **Web API가 타이머를 관리한다.** Web API는 2000ms 동안 타이머를 관리하며 대기합니다. 이 과정에서 자바스크립트 엔진은 다른 작업을 계속 처리할 수 있습니다.

4. **타이머가 만료되면 Web API가 이벤트 루프에 알린다.** 타이머가 만료되면, Web API는 이벤트 루프에 콜백 함수를 처리할 준비가 되었음을 알립니다.

5. **이벤트 루프가 콜백을 태스크 큐에 넣는다.** 이벤트 루프는 콜백을 **매크로 태스크 큐(Macro Task Queue)**에 추가합니다.

### 사례 2 - `fetch`

`fetch`도 Web API의 대표적인 예입니다. 이 코드를 통해 **비동기 네트워크 요청**이 어떻게 처리되는지 살펴보겠습니다.

```js
fetch('https://example.com/todos.json') 
  .then(response => {
    if (!response.ok) {
      throw new Error('Network response was not ok');
    }
    return response.json();
  })
  .then(data => {
    console.log(data);
  })
  .catch(error => {
    console.error('There was a problem with the fetch operation:', error);
  });
```

#### 실행 과정에서 Web API의 역할
1. **콜 스택에 fetch 함수가 push된다.** 자바스크립트 엔진이 fetch()를 호출하면, 콜 스택에 fetch()가 추가됩니다.

2. **fetch 함수의 실행은 Web API에 위임된다.** fetch() 함수는 브라우저의 Web API로 네트워크 요청을 위임합니다. Web API는 네트워크 요청을 처리하는 동안 네트워크 스레드에서 작업을 진행하고, 자바스크립트 엔진은 다른 작업을 계속할 수 있습니다. 이때 fetch()는 콜 스택에서 제거됩니다.

3. **네트워크 요청이 완료되면 Web API는 then 콜백을 태스크 큐에 추가한다.** 네트워크 응답이 완료되면, Web API는 then으로 전달된 Promise 콜백을 **마이크로 태스크 큐(Micro Task Queue)**에 추가합니다.

세심하신 분들은 알아차렸겠지만, setTimeout의 콜백 함수와 Promise의 then 콜백은 **각기 다른 큐에서 대기**하게 됩니다. 비동기 작업이 어떻게 처리되는지에 따라 그 콜백이 대기하는 태스크 큐도 달라지기 때문입니다. 이번에는 이러한 차이를 더 자세히 살펴보겠습니다.

## 2️⃣ 태스크 큐

**태스크 큐**는 비동기 작업의 콜백 함수(태스크)가 대기하는 큐입니다. 비동기 작업의 종류에 따라 각기 다른 큐에서 대기하게 되며, 대기 중인 태스크들은 이벤트 루프에 의해 순차적으로 콜 스택에서 처리됩니다.

### 1. 매크로 태스크 큐(Macro Task Queue)
`setTimeout`, `setInterval`, `파일 I/O 작업`, `DOM 이벤트`등 비교적 크고 독립적인 작업이 대기하는 큐입니다. 

### 2. 마이크로 태스크 큐(Micro Task Queue)
Promise의 `then()`, `MutationObserver` 등과 같이 작고 빠르게 처리되어야 하는 작업이 대기하는 큐입니다.

## 3️⃣ 이벤트 루프

앞에서도 계속 언급되었던 **이벤트 루프**는 자바스크립트 런타임에서 비동기 작업을 처리하고 콜백을 실행하기 위한 핵심 메커니즘입니다. 

**콜 스택이 비어 있을 때, 태스크 큐의 종류와 우선순위**에 따라 이를 처리합니다. 

#### 1. **마이크로 태스크 큐** 
처리 우선순위가 가장 높으며, **큐가 텅 빌 때까지 계속해서 호출됩니다**. 또한 `queueMicrotask()`를 통해 개발자가 마이크로 태스크를 추가할 수도 있는데, 이 또한 다음 태스크 실행 전에 모두 실행됩니다
> **‼️ 경고: 재귀적으로 마이크로 태스크를 추가할 때는 주의가 필요합니다.**
> 마이크로 태스크는 다음 태스크 실행 전 모두 처리되므로 이벤트 루프가 끝없는 마이크로 태스크 처리 루프에 빠질 위험이 있습니다. 이 경우 성능 문제를 야기합니다.

#### 2. **매크로 태스크 큐**
처리 우선순위가 가장 낮으며 한 번에 하나씩만 처리합니다. 즉, **콜 스택에 동기 태스크가 없으며, 마이크로 태스크 큐나 애니메이션 프레임 큐가 비어있어야** 이벤트 루프가 처리합니다. 

> 🙋‍♀️ **그렇다면 setTimeout의 지연시간을 0초로 설정해도 실제 지연 시간은 더 길어질 수 있나요?**

-> 맞습니다. setTimeout의 콜백은 매크로 태스크 큐에 들어가므로 처리해야 할 마이크로 태스크가 있거나, 동기 코드가 있으면 실제 지연시간이 0초가 될 수 없습니다. 따라서 setTimeout의 두 번째 인자는 실제 지연시간이 아닌 **최소 지연시간**입니다.

## 🤔 간단한 문제!

이러한 우선순위에 따라 아래 코드의 console이 어떻게 찍히는지 예측해 보세요.

```js
console.log("Frame start");

setTimeout(() => {
  console.log("Timeout");
  Promise.resolve().then(() => {
    console.log("Promise inside Timeout");
  });
}, 0);

Promise.resolve().then(() => {
  console.log("Promise 1");
}).then(() => {
  console.log("Promise 2");
});

console.log("Frame end");
```

> **정답**
Frame start
Frame end
Promise 1
Promise 2
Timeout
Promise inside Timeout
